# Testing the hub — methodology

Как мы тестируем telegram-tmux-channels. Проект интерактивный и завязан на **живую LLM** и
на **рендер в Telegram**, поэтому классические код-e2e покрывают не всё — центр тяжести смещён
в сторону «прогнать реального бота и посмотреть глазами + в логи». Методология заимствована из
ресёрча в `agentek-console` (planner→generator→healer, слоистая пирамида, fault-injection,
bug→guard), адаптирована под бота без веб-UI.

## Слои (тестируем на самом нижнем, который ловит баг)

1. **Unit** (`tests/*.test.ts`, `bun test`) — чистая stdlib-логика: парсеры пейна
   (`parseOpsCommand`, `paneDigest`, `parseCompaction`, `paneIsWorking`…), персист-репозиторий,
   роутинг ключей. Дёшево, детерминированно — сюда кладём всё, что можно.
2. **Integration (real bot in Docker)** — тонкий слой «критических путей»: гоняем настоящий хаб
   в контейнере `clod-tmux`, драйвим бота своим Telegram-аккаунтом, наблюдаем через логи/API.
   Здесь живут сценарии, которые нельзя проверить юнитом (интерактив, tmux, доставка в TG).
3. **Telegram-Web (Playwright)** — **самый тонкий** слой: визуальная проверка рендера глазами
   (дубли сообщений, тред/якоря, склейка, кнопки, переходы состояний). Ловит то, что Bot-API
   не отдаёт. Только по критическим фичам, не на каждый чих.
   Веб на **/a/** (WebA), профиль `~/.cache/ms-playwright-mcp/telegram-profile` (залогинен под
   аккаунтом-драйвером). Быстрый снимок — headless: `google-chrome --headless=new
   --user-data-dir=<PROF> --virtual-time-budget=14000 --screenshot=out.png
   "https://web.telegram.org/a/#<chatId>"` (перед запуском `rm -f <PROF>/SingletonLock`).
   ⚠️ **Гоча headless (подтверждено 2026-07-18):** ленивый контент (reply-quote превью,
   кастом-эмодзи, медиа) грузится не мгновенно — короткий снимок ловит «Loading…», хотя в живом
   клиенте всё резолвится. Это артефакт таймингов, НЕ баг. Фикс авто-снимка: ждать дольше (headless
   `--virtual-time-budget` мало для reply-превью, дай 20–30с), лучше — **MCP-браузер + `browser_wait_for`**
   на нужный текст перед `browser_take_screenshot`. Снять RDP-дисплей headless НЕ выходит (`grim` →
   «no wl_output», `gnome-screenshot` → пустой X11-fallback; mutter/RDP не отдаёт wlr-screencopy), так что
   **headed-окно — только для живых глаз Ромы**. Правило: не судить «баг рендера» по одному быстрому кадру.
   ⚙️ **Пин профиля для MCP-браузера:** чтобы `plugin:playwright` (Microsoft `@playwright/mcp`) поднимался
   на нашем залогиненном профиле, в его `.mcp.json` добавлены аргументы
   `--browser chrome --user-data-dir /home/user/.cache/ms-playwright-mcp/telegram-profile`
   (файлы: `~/.claude/plugins/cache/claude-plugins-official/playwright/unknown/.mcp.json` и зеркало в
   `marketplaces/.../external_plugins/playwright/.mcp.json`). Применяется **только при ПОЛНОМ рестарте
   сессии claude** (`/mcp reconnect` переиспользует старую команду). Может слететь при апдейте плагина —
   тогда прописать заново. Логин раскладывается в профиль разово (headed-chrome на RDP-дисплее → вход
   под аккаунтом-драйвером; K и A логинятся раздельно, нам нужен /a/).

Толстый e2e — антипаттерн. Если баг можно поймать юнитом — пишем юнит, а не сценарий.

## Where to look — поверхности наблюдения

Не верим самоотчётам агента/бота — только ground truth:

| Поверхность | Как снять | Что ловит |
|---|---|---|
| Telegram API | telegram-mcp `list_messages` / `list_inline_buttons` / `get_message_reactions` | что реально отправлено/кнопки/реакции |
| Telegram Web | Playwright-браузер (скрин/снапшот) | рендер: дубли, тред, склейка, layout |
| `screenlog.jsonl` | `TELEGRAM_DEBUG_LOG=1`; читать хвост | таймлайн хаба `screen`/`tg_in`/`tg_out` с ФИНАЛЬНЫМ текстом |
| hub-лог | `/tmp/hub-*.log` (контейнер) / `journalctl --user -u telegram-hub` (прод) | старт, ошибки, инъекции команд |
| tmux pane | `tmux capture-pane -p -t <pane>` | что реально в TUI сессии |
| `hub-state.json` | `cat ~/.claude/channels/telegram/hub-state.json` | персист: pending/fallback/permissions/pickers |
| `bindings.json` | там же | роутинг топик→папка |

## Per-case template (канон — копируем в каждый feature-док)

Каждый кейс — четыре поля:

- **Шаги** — действие пользователя ИЛИ инъекция сбоя (убить сокет/пейн, рестарт в середине хода, два окна).
- **Ожидаемо** — корректное поведение, которое подтверждаем.
- **Fail-режим** — известный/подозреваемый слом, который ловим (со ссылкой на корень, если есть).
- **Смотреть** — поверхность наблюдения из таблицы выше.

Теги на кейсах (для планирования параллельных прогонов):
- `[СПЕЦ]` — нужна спец-инъекция (рестарт хаба, убить пейн, битый STT, нет chrome).
- `[2-ЧАТА]` — нужно ≥2 контекста/биндинга.
- `[ЭКСКЛЮЗИВ]` — мутирует общий стейт → гонять изолированно, ломает соседей.

Каждый feature-док начинается с **Цель / Предусловия**, дальше блоки кейсов, в конце — **Лог прогона**
(дата, что нашли, статус).

## Fault-injection — первый класс, не «на потом»

Самый ценный слой для интерактивной resumable-системы (там же и живут баги — дубли/потери на
реконнекте). Обязательный набор для Mux:
рестарт хаба (в середине хода / пермишена / открытого пикера), убитый tmux-пейн (💀), двойная/быстрая
отправка, `/esc` посреди хода (дренаж очереди), два чата на одну папку, сбой STT, нет chrome
(`/screen`→`/last`), несколько `/screen`/`/last` (one-per-pane), фаззи слэш-инъекция.

## Процесс на фичу (planner → runner, две фазы)

1. **План** — субагент/скилл `feature-test` исследует фичу по коду+README, пишет сценарии в
   `features/<фича>.md` по шаблону (позитив + негатив + швы + fault-injection).
2. **Прогон** — поднять контейнер `clod-tmux`, прогнать сценарии (драйв бота → наблюдение по
   поверхностям → Telegram-Web визуал), записать результат в «Лог прогона».
3. **Фикс + взгляд сеньора** — баг чиним по корню; если архитектура позволяет багу случаться,
   заворачиваем в переиспользуемый хелпер/паттерн (одна точка, которую нельзя обойти), а не точечно.
   Каждый фикс → строка в `regression-checklist.md` + по возможности guard-юнит.
4. Обновить статус в `feature-map.md`.

## 👻 Фантомный ввод: строка input — НЕ пользовательский ввод

Claude Code показывает в пустом поле ввода **подсказку следующего промпта**, сгенерированную из
истории сессии (а также generic `Try "fix lint errors"`). В снимке пейна это выглядит ровно как
набранный юзером текст, хотя **никто его не отправлял** (2026-07-20: принял такие подсказки за
«кто-то печатает в сессию», поднял ложную тревогу — в `tg_in` этих сообщений не было вовсе).

Следствия: (1) не судить «юзер что-то ввёл» по строке `❯ …` — сверяться с `tg_in` в `screenlog.jsonl`;
(2) `/last`-дайджест и pane-детекторы могут показать этот фантом как реальный ввод — кандидат в
негативный кейс.

✅ **Источник погашен (2026-07-20):** подсказки выключены штатно — в `~/.claude/settings.json`
`env.CLAUDE_CODE_ENABLE_PROMPT_SUGGESTION = "false"` (переменная реально есть в бинаре 2.1.215
рядом с прочими `CLAUDE_CODE_ENABLE_*`). Проверено на живом пейне: строка ввода стала пустой `❯`,
исчез и generic `Try "fix lint errors"`. Подхватывается сессиями, стартовавшими ПОСЛЕ правки —
уже открытые пейны продолжают показывать подсказки до рестарта. Правило (1) всё равно оставляем:
на чужих/старых конфигах фантом возможен.

## ⛔ Никаких «✅ по наблюдению»

Статус ✅ ставится **только** за реальный прогон кейса. «В моих прогонах оно всплывало и
прожевалось» — это НЕ тест, это совпадение: так был помечен PB5 (авто-ack стартовых промптов), и
именно там потом висел прод (нажатие Enter проглатывалось, повтора не было — 2026-07-20). Если
кейс не гонялся отдельно и негативно — он ⬜, а не ✅. Особенно это касается «оно же и так
работает» — такие места ломаются молча.

## Bug → guard (дисциплина)

Починил баг → строка в `regression-checklist.md` (симптом / корень / фикс-коммит / guard-тест).
Вернётся баг — падает `bun test`, а не пользователь.

## Repeatability

Всё воспроизводимо: контейнер поднимается из `docker-compose.yml`, драйв — телеграм-аккаунтом,
состояние переживает рестарт (Stage 1–3), новая сессия видит доки и продолжает с `feature-map.md`.
