# Feature: binding-routing

**Цель.** `/bind <папка>` привязывает топик/DM к папке проекта, `/unbind` отвязывает (и убивает
поднятую хабом tmux-сессию), `/allow <id>` даёт доступ не-админу в рамках ОДНОГО биндинга. Роутинг:
ключ чата (`chatId/topicId` | `chatId` | `dm:userId`) → `bindings.json` → папка → живые сессии с этим
cwd. `bindings.json` правится хабом и подхватывается на лету.

**Код.** ops `bind`/`unbind`/`allow` в `hub.ts`, `bindings.ts` (`messageKey`/`keyToTarget`),
`registry.ts` (`keysForDir`/`resolveProjectDir`), `ownKeys`, `connsForBinding`.

**Предусловие.** Контейнер; топик 134 не забинден (исключён из авто-бинда), топики 3 и 129 оба
смотрят в `/home/user/projects/sandbox` — идеально для проверки роутинга при общей папке.

- Ожидаемо: в `bindings.json` появился ключ `-1004355407865/134` с `dir`; бот подтвердил.

**BR2. 🎯 Роутинг при ОБЩЕЙ папке: ответ приходит в свой топик.** `[СПЕЦ]`
- Шаги: топики 3 и 129 смотрят в одну папку; написать в 129.
- Ожидаемо: ответ агента приходит в **129**, не в 3 (и наоборот) — ключ решает, а не папка.
- Fail-режим: ответ уехал в соседний топик с тем же cwd (классический слом `keysForDir`).
- Смотреть: чат обоих топиков (`reply_to`/thread).

**BR3. `/unbind` — снимает биндинг и убивает tmux-сессию.** `[СПЕЦ]`
- Шаги: `/unbind` в 134.
- Ожидаемо: ключ исчез из `bindings.json`; tmux-сессия этого биндинга убита.
- Смотреть: `bindings.json`, `tmux ls`.

## Негатив

**BR4. Не-админ не может `/bind`/`/unbind`/`/allow`.** ⬜
- Нужен второй аккаунт (не-админ) — в текущем харнессе нет.

**BR5. `/allow <id>` даёт доступ только к своему биндингу.** ⬜
- Тоже требует второго аккаунта.
