# Feature: core-messaging

**Цель.** Входящее из Telegram → в сессию; агент отвечает `reply`/`react`/`edit_message`. В чат
уходит ТОЛЬКО то, что агент шлёт через egress-тулы (не весь транскрипт). Сообщения, присланные
пока сессия поднимается, придерживаются (👌) и досылаются по коннекту.

**Код.** `handleInbound`, `doReply`/`doReact`/`doEdit`/`doDownload`, `enqueueForTopic`/`flushQueued`,
`clearPendingAnswer`, `ownKeys`/роутинг ключа.

**Предусловие.** Контейнер, биндинг топик 3.

---

## Позитив

**CM1. Входящее → агент → `reply` в топик.** ✅ (покрыто множеством прогонов)
- Ссылка: любой прогон, где сообщение доходило до агента и он отвечал в топик (RP3, skills, и т.д.).

**CM2. `react` — реакция на сообщение.**
- Шаги: попросить агента поставить реакцию (`react`).
- Ожидаемо: на сообщении появляется эмодзи-реакция.
- Смотреть: API (`get_message_reactions`), чат.

**CM3. `edit_message` — правка своего сообщения.**
- Шаги: агент шлёт `reply`, затем `edit_message`.
- Ожидаемо: текст сообщения обновился на месте.
- Смотреть: API (`list_messages` — текст изменился).

## Негатив / швы / edge

**CM4. 🎯 Очередь при поднятии сессии (hold → 👌 → досыл).** `[СПЕЦ]`
- Шаги: `/stop` (сессия вниз) → прислать обычное сообщение (не ops).
- Ожидаемо: на сообщении реакция 👌 (held), хаб авто-поднимает сессию, `flushQueued` доставляет
  сообщение агенту после коннекта; агент отвечает.
- Fail-режим: сообщение потеряно (не доставлено), или «сессия не поднялась вовремя».
- Смотреть: API (реакция 👌), пейн (агент получил), чат.

**CM5. Только `reply` уходит в чат, не транскрипт.**
- Шаги: дать агенту задачу с рассуждением, но без `reply` в конце.
- Ожидаемо: болтовня агента НЕ уходит в чат (только reply-fallback добор, если совсем нет egress —
  это отдельная фича `reply-fallback`).
- Смотреть: чат (нет спама транскриптом).

**CM6. Роутинг: ответ уходит строго в свой топик.**
- Шаги: сообщение в топик 3.
- Ожидаемо: reply агента приходит в топик 3 (`reply_to: 3`), не в General/другой топик.
- Смотреть: API (`reply_to`/thread).
