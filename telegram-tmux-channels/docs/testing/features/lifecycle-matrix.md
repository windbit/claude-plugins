# Feature: lifecycle-matrix (tmux ↔ claude ↔ папка ↔ топик)

**Цель.** Сквозная проверка жизненного цикла: как биндинг, tmux-сессия, беседа claude и топик
Telegram переживают `/bind` `/unbind` `/delete` `/resume` `/new` `/restart` `/stop`, удаление
топика, рестарт хаба и ребут — во всех достижимых комбинациях. Плюс **удобство**: понятно ли из
текста, ЧТО произошло (id/имя/папка), нет ли лишних шагов, не теряются ли сообщения.

**Не дублирует:** `binding-routing` (роутинг ключ→папка, доступ), `session-lifecycle` (одиночные
`/resume` `/new` `/restart`), `trusted-groups` (авто-бинд, режимы), `restart-persistence`
(Stage 1–3 персист), `death-notice` (💀). Этот док — про **переходы между ними** и стыки.

**Код.** `hub.ts`: `spawnSession` (1916), `stopLiveSessions` (1899), `reviveBoundSessions` (1966),
`teardownBinding` (1989), `onTopicGone` (2036), `runAutoTopic`/`beginTopicSession`/`handleLateTopic`
(2104–2192), `handleInbound` (2201), `handleOps` (2450), `connsForBinding` (1674), `forkRiskPids`
(1711), `sessionName` (1692), `verifyClaimedKeys` (1560), `syncSessionId` (1580), `learnCmdline`
(1634); `tmux-ops.ts` (`buildLaunch`, `stopSession`, `restartSession`), `session-id.ts`
(`jsonlMtimes`, `captureNewSessionId`, `recentSessions`), `registry.ts`, `dir-resolve.ts`,
`trusted-groups.ts`, `known-chats.ts`.

**Предусловия.** Контейнер `clod-tmux`, тест-бот `@clod_tmux_lab_bot`, группа `-1004355407865`
(форум), `trusted-groups.json` на неё, `TELEGRAM_DEBUG_LOG=1`. Для группы J — второй,
не-админский аккаунт (в текущем харнессе нет).

## 1. Матрица состояний и переходов

Оси состояния (достижимые комбинации):

| Ось | Значения |
|---|---|
| Биндинг **B** | `B0` нет · `B1` есть, без `sessionId` · `B2` есть, с `sessionId` |
| tmux/claude **T** | `T0` нет tmux-сессии · `T1` tmux есть, пейн = шелл (claude мёртв) · `T2` claude жив и подключён (stub) · `T3` claude жив, но НЕ подключён (запущен руками/вне хаба) |
| Режим топика **M** | `M0` ручной `/bind` · `M1` trusted+folder · `M2` trusted+worktree(+хук) |
| Папка **D** | `D0` своя · `D1` общая на 2 биндинга · `D2` worktree |
| Источник **S** | `S0` админ · `S1` allow-юзер · `S2` посторонний |

Переходы (строка = состояние, клетка = ожидаемый исход + кейс). `—` = недостижимо.

| B/T | обычное сообщение | `/resume` | `/resume <id>` | `/new` | `/restart` | `/stop` | `/unbind` · `/delete` |

группа J. Недостижимо: `B0/T2` с ключом (сессия без биндинга не заявляет ключ — `verifyClaimedKeys`
их отбросит), `B*/T3` в worktree-папке до её создания.

## 2. Кейсы

Формат: **Шаги / Ожидаемо / Fail-режим / Смотреть**. `P0` ломает работу или теряет данные,
`P1` путает, `P2` косметика. 🤖 = гоняется в докер-харнессе через telegram-mcp, 👁 = нужен глаз,
👤 = нужен второй аккаунт, ✋ = ручное действие в клиенте (нет MCP-метода).

### A. Новый топик → бинд → первый запуск

**LM1. Новый топик trusted-группы с ОДНИМ режимом.** `[СПЕЦ]` P1 🤖
- Шаги: `create_forum_topic` в группе с `modes:["folder"]`.
- Ожидаемо: кнопки режима + «✏️ Своя папка», биндинг и спавн — только после тапа.
- Fail-режим: молчаливый бинд в дефолтную папку + автостарт (регрессия 2026-07-23).
- Смотреть: чат топика, `bindings.json`, `tmux ls`.

**LM2. «✏️ Своя папка» → путь.** P1 🤖
- Шаги: тап `topicdir:` → прислать `sandbox` (и отдельно — абсолютный путь, и `~/projects/x`).
- Ожидаемо: биндинг на указанную папку, сессия там же, `pendingTopics` очищен.
- Fail-режим: биндинг в дефолтную папку группы; повторный вопрос про папку.
- Смотреть: `bindings.json`, пейн (`pwd`), чат.

**LM3. Ответ «своя папка» — несуществующий путь.** P1 🤖
- Шаги: в ответ на «пришли папку» отправить `/nope/nope`.
- Ожидаемо: «⚠️ Не похоже на папку: …», топик ОСТАЁТСЯ в ожидании, второй ответ проходит.
- Fail-режим: топик выпал из `pendingTopics` → дальше сообщения дропаются как unbound.
- Смотреть: чат, hub-лог (`drop (unbound)`), `bindings.json`.

**LM4. Сообщение до выбора режима не теряется.** (=TG4) P0 🤖
- Шаги: новый топик → НЕ нажимая кнопку, написать задачу → выбрать режим.
- Ожидаемо: 👌 на сообщении, после подъёма сессии `flushQueued` доставляет его агенту.
- Fail-режим: 👌 стоит, сообщение не доставлено (очередь `queuedMessages` — только в памяти).
- Смотреть: реакции (`get_message_reactions`), `screenlog.jsonl` (`screen` с текстом задачи), ответ агента.

**LM5. Гонка: опс-команда сразу после создания топика.** `[СПЕЦ]` P1 🤖
- Шаги: новый топик → сразу `/status` (или `/resume`) до тапа по режиму.
- Ожидаемо: команда отрабатывает И выбор режима не теряется (или хаб честно говорит, что выбор снят).
- Fail-режим: `handleInbound:2226-2227` безусловно делает `pendingTopics.delete`/`pendingModeChoice.delete` →
  кнопки мертвы («Уже выбрано или устарело»), топик остаётся без биндинга, повторного вопроса нет.
- Смотреть: чат (нажать кнопку после), hub-лог, `bindings.json`.

**LM6. Ручной `/bind` в не-trusted топике → первый запуск.** P1 🤖
- Шаги: `/bind sandbox` в чистом топике → тап «🆕 Новая сессия».
- Ожидаемо: «🔗 Привязано», кнопки, потом «🆕 Запускаю сессию» (НЕ «заново» — фикс a85313f),
  `sessionId` дописан в биндинг за ≤60с (`captureNewSessionId`).
- Fail-режим: «🚀 Запускаю заново» на первом старте; `sessionId` не выучен → следующий revive `--continue`.
- Смотреть: чат, `bindings.json` (появление `sessionId`), hub-лог `learned sessionId`.

**LM7. `/bind` ПОВЕРХ существующего биндинга.** `[ЭКСКЛЮЗИВ]` **P0** 🤖
- Шаги: в живом биндинге (есть `sessionId`, выученный `cmdline`, worktree с `hookBranch`) — `/bind <та же папка>`.
- Ожидаемо: беседа и режим сохраняются; текст явно говорит «было X → стало Y».
- Fail-режим: `hub.ts:2526` пишет `reg[key] = { dir, allow }` → **теряются `sessionId`, `cmdline`,
  `hookBranch`**: беседа осиротела (revive пойдёт `--continue`), worktree после `/unbind` не приберётся.
- Смотреть: `bindings.json` до/после, потом `/unbind` (проверить, что хук не сработал).

### B. Переключение бесед (`/resume`)

**LM8. `/resume` без аргумента, мёртвая сессия, ≥2 бесед.** P1 🤖
- Шаги: `/resume` в биндинге, где `recentSessions ≥ 2`.
- Ожидаемо: «⏪ Какую сессию поднять?» с датой+снипетом; тап `rs:` → `--resume <id>`, `sessionId` записан.
- Fail-режим: мгновенный старт не той беседы; кнопки с пустыми лейблами (`firstUserText` не распарсил).
- Смотреть: `list_inline_buttons`, пейн (argv), `bindings.json`.

**LM9. `/resume`, ровно одна беседа.** P2 🤖 — без пикера, сразу `--resume`/`--continue`.

**LM10. `/resume` в папке БЕЗ истории бесед.** **P1** 🤖
- Шаги: `/bind <пустая папка>` → `/resume` (не трогая кнопки).
- Ожидаемо: хаб понимает, что возобновлять нечего, и стартует новую (или говорит об этом).
- Fail-режим: `spawnSession(mode='resume', sessionId=undefined)` → `buildLaunch` даёт `--continue`
  (`tmux-ops.ts:211`), claude падает «No conversation found» и выходит; в чате висит «🚀 Возобновляю»,
  пейн — пустой шелл, уведомления нет.
- Смотреть: пейн (`capture-pane`), чат, hub-лог.

**LM11. `/resume` при ЖИВОЙ сессии — нативный TUI-пикер.** P1 🤖
- Шаги: живая сессия → `/resume` → тап по строке.
- Ожидаемо: список = ровно строки TUI, тап переключает беседу БЕЗ рестарта процесса; следующий ход
  прогоняет `syncSessionId` и биндинг получает новый id.
- Fail-режим: «Не попал по курсору»; переключились, а `bindings.json` держит старый id → следующий
  revive поднимет прошлую беседу (корень бага 2026-07-20).
- Смотреть: `list_inline_buttons`, пейн, `bindings.json` ПОСЛЕ первого хода, hub-лог `sessionId synced`.

**LM12. `/resume <полный id>` при живой сессии.** P1 🤖
- Ожидаемо: `stopLiveSessions` гасит текущую (не форк), спавн с `--resume <id>`, ровно один claude в пейне.
- Fail-режим: форк беседы (два claude на один id); «Не смог остановить» при висящем `Exit anyway?`.
- Смотреть: пейн (`pgrep`/`tmux list-panes`), argv, чат.

**LM13. `/resume <префикс>`: 1 / 0 / ≥2 попаданий.** P1 🤖
- Ожидаемо: 1 → подъём; 0 → «Нет сессии <x> в <папке>»; ≥2 → «подходит N сессиям — уточни».
- Fail-режим: аргумент молча игнорируется (баг 2026-07-23); регистр (`ABC` vs `abc`) не матчится.
- Смотреть: чат, hub-лог, argv в пейне.

**LM14. `/resume <id>` при УБИТОМ tmux-сервере.** `[СПЕЦ]` P1 🤖
- Шаги: `tmux kill-server` → `/resume <префикс>`.
- Ожидаемо: tmux создан заново, argv содержит нужный id.
- Смотреть: `tmux ls`, пейн.

**LM15. `/resume <id>` беседы соседнего топика (общая папка).** `[2-ЧАТА]` P1 🤖
- Шаги: топики A и B на одной папке → в A `/resume <id беседы B>` при живой сессии B.
- Ожидаемо: либо отказ («эту беседу ведёт топик B»), либо честное предупреждение — но не тихий форк.
- Fail-режим: `jsonlMtimes(dir)` не знает про владельца → A поднимает беседу B, `forkRiskPids` не спасает
  (сессия B **tracked**, значит исключена из проверки) → два claude на один id, разъезд истории.
- Смотреть: оба топика, `bindings.json` (у обоих один `sessionId`), пейны.

**LM16. Лост-апдейт `bindings.json` на долгой операции.** `[СПЕЦ]` P1 🤖
- Шаги: в топике A `/resume <id>` (внутри — `stopSession`, десятки секунд); пока идёт — в топике B
  сделать ход (сработает `syncSessionId`/`learnCmdline`).
- Ожидаемо: обе записи в `bindings.json` сохранены.
- Fail-режим: `handleOps` читает `reg` на 2454 и пишет `saveBindings(reg)` на 2598 → изменения B затёрты.
- Смотреть: `bindings.json` до/после, hub-лог (`sessionId synced` для B, потом откат).

### C. `/new` поверх существующего

**LM17. `/new` при мёртвой сессии.** P1 🤖
- Ожидаемо: свежая беседа, «🚀 Запускаю заново» (если `sessionId` был), новый id записан в биндинг.
- Fail-режим: `captureNewSessionId` не поймал файл за 60с → биндинг держит СТАРЫЙ id → следующий
  revive поднимет прошлую беседу.
- Смотреть: `bindings.json`, `~/.claude/projects/<slug>/`, hub-лог.

**LM18. `/new` при ЖИВОЙ сессии.** **P1** 🤖
- Ожидаемо: новая беседа (со стопом текущей) или явный отказ с готовой кнопкой.
- Fail-режим: `hub.ts:2811` отвечает «⚙️ Сессия уже подключена… Используй /restart или /compact» —
  про `/new` ни слова, начать новую беседу из чата нечем (нужно `/stop` → кнопка).
- Смотреть: чат, пейн (беседа не сменилась).

**LM19. `/new` в двух топиках одной папки одновременно.** `[2-ЧАТА][ЭКСКЛЮЗИВ]` P1 🤖
- Ожидаемо: каждый биндинг получает СВОЙ новый `sessionId`.
- Fail-режим: `captureNewSessionId` берёт «первый файл, которого не было в снимке» → топики могут
  обменяться id, дальше `/resume` поднимет чужую беседу.
- Смотреть: `bindings.json`, содержимое обоих `.jsonl` (первое сообщение).

### D. `/restart` · `/stop`

**LM20. `/restart` живой сессии.** P1 🤖
- Ожидаемо: graceful `C-c`→`/exit`→relaunch, тот же id, в команде relaunch есть
  `TELEGRAM_BINDING_KEYS=` (иначе статус-посты замолкают, а пикеры уедут в чужой топик).
- Fail-режим: relaunch без env-префикса; `--resume` превратился в bare `--resume` (пикер без пользователя).
- Смотреть: пейн (набранная команда), hub-лог `restart: relaunch …`, статус-пост после перезапуска.

**LM21. `/restart` (и `/stop`) мёртвой сессии.** (=SL6) **P1** 🤖
- Ожидаемо: `/restart` поднимает как `/resume` (очевидное действие).
- Fail-режим: `hub.ts:2652-2655` — «⚠️ Нет живой сессии. Попробуй /resume»: лишнее действие,
  а плашка 💀 при этом обещает, что `/resume` поможет — значит и `/restart` должен.
- Смотреть: чат.

**LM22. `/stop` живой сессии.** P1 🤖
- Ожидаемо: «🛑 Останавливаю…» → «Сессия остановлена. Что дальше?» + кнопки; 💀-плашки НЕТ
  (`expectedDisconnect`).
- Fail-режим: и «остановлена», и 💀 подряд; кнопки «что дальше» без `rs:`-строк.
- Смотреть: чат (порядок сообщений), `list_inline_buttons`.

**LM23. `/stop` при висящем «Exit anyway?».** `[СПЕЦ]` P1 🤖
- Шаги: запустить фоновый шелл в сессии → `/stop`.
- Ожидаемо: конфирм приезжает кнопками (picker-мост), 10с грейс, иначе Enter; процесс мёртв.
- Fail-режим: висит вечно; «Процесс не умер» при фактически мёртвом pid.
- Смотреть: пейн, чат, `alive(pid)`.

**LM24. `/restart` посреди хода с очередью.** `[СПЕЦ]` P1 🤖
- Шаги: дать длинную задачу + дослать 2 сообщения → `/restart`.
- Ожидаемо: ход прерван, очередь не «догоняет» после перезапуска, история цела (`--resume <тот же id>`).
- Fail-режим: после relaunch агент доигрывает старую очередь; потеря id.
- Смотреть: `screenlog.jsonl`, транскрипт `.jsonl`, пейн.

### E. `/unbind` · `/delete`

**LM25. `/unbind` — отчёт называет ЧТО снесли.** P1 🤖
- Ожидаемо: ключ исчез, tmux убит, текст содержит `#topicId` + имя топика + папку + `sessionId`.
- Fail-режим: обезличенное «Отвязано» (баг 2026-07-23); имя топика неизвестно, потому что
  `known-chats.json` знает его только из `forum_topic_created`/`_edited` — топик, впервые увиденный
  по обычному сообщению, останется без имени.
- Смотреть: чат, `bindings.json`, `tmux ls`, `known-chats.json`.

**LM26. `/unbind` worktree-биндинга без хука.** P1 🤖
- Ожидаемо: `removePlainWorktree` сносит связанный worktree и говорит об этом; на main-checkout —
  НЕ сносит и молчит.
- Fail-режим: удалён основной чекаут; «Удаление worktree не удалось» на обычной папке.
- Смотреть: `git worktree list`, файловая система, чат.

**LM27. `/unbind` при ЖИВОМ claude.** P1 🤖
- Ожидаемо: tmux убит вместе с процессом, 💀-плашки нет (топик уже отвязан → `notifyUnexpectedDeath`
  не находит биндинга).
- Fail-режим: осиротевший claude без tmux; 💀 в отвязанный топик.
- Смотреть: `pgrep claude`, чат.

**LM28. `/delete` в топике (с биндом и без).** P1 🤖
- Ожидаемо: teardown + `deleteForumTopic`, один отчёт в General с id/именем/папкой/сессией; без бинда —
  «бинда не было» + топик удалён.
- Смотреть: General, `bindings.json`, `list_topics`.

**LM29. `/delete` без права `can_delete_messages`.** `[СПЕЦ]` P1 🤖
- Ожидаемо: явно сказано, что биндинг снят, а топик остался (и что следующий месседж в нём заново
  предложит режим).
- Fail-режим: «Отвязано» + «Топик не удалён» без вывода → юзер думает, что всё сломалось; в trusted-группе
  следующий месседж внезапно запускает мастер режима.
- Смотреть: чат, права бота, `bindings.json`.

### F. Топик удалён руками

**LM30. Удаление топика при живой сессии.** `[СПЕЦ]` P1 ✋
- Шаги: удалить топик в клиенте → дождаться `typing`/ответа агента.
- Ожидаемо: `isThreadGoneError` → `onTopicGone` → teardown, отчёт в General, tmux убит.
- Fail-режим: сессия живёт вечно, отчёта нет; двойной teardown (гвард `tearingDown`).
- Смотреть: General, `bindings.json`, `tmux ls`, hub-лог `topic gone`.

**LM31. Удаление топика, пока хаб лежит.** `[СПЕЦ]` **P0** ✋
- Шаги: остановить хаб → удалить топик → `tmux kill-server` → поднять хаб.
- Ожидаемо: биндинг мёртвого топика прибран, зомби-сессия не поднимается.
- Fail-режим: `reviveBoundSessions` (1971-1983) спавнит tmux+claude, а его `say` (1977-1980) шлёт
  напрямую с `.catch(() => {})` — **мимо** обработки `isThreadGoneError` (ср. `typing` на 84-85) →

- Смотреть: `bindings.json`, `tmux ls`, `pgrep claude`, hub-лог.

### G. Рестарт хаба и ребут

**LM32. Рестарт хаба при живых tmux.** `[СПЕЦ]` P1 🤖
- Ожидаемо: `hasTmuxSession` → revive пропускает, стабы переподключаются сами, «Запускаю» не дублируется,
  ложных 💀 нет.
- Fail-режим: повторный спавн поверх живой сессии; шквал 💀.
- Смотреть: чат, hub-лог `boot-revive`, `tmux ls`.

**LM33. Ребут / `tmux kill-server` → revive всех биндингов.** `[СПЕЦ][ЭКСКЛЮЗИВ]` P1 🤖
- Ожидаемо: каждый биндинг поднят со своим `--resume <id>`, стаггер 3с, сообщение — в свой топик.
- Fail-режим: revive идёт до появления `botUsername`/сети → сообщения теряются; биндинги с мёртвой
  папкой валят цикл (`spawnSession` ловит ошибку — проверить, что цикл не прерывается).
- Смотреть: все топики, hub-лог, `tmux ls`.

**LM34. Рестарт хаба между запуском и выучиванием id.** `[СПЕЦ]` P1 🤖
- Шаги: `/new` → убить хаб через 3с (до `captureNewSessionId`) → поднять.
- Ожидаемо: `sessionId` дописывается на первом же ходе через `syncSessionId`.
- Fail-режим: биндинг навсегда без id → любой revive идёт `--continue` (не та беседа при общей папке).
- Смотреть: `bindings.json`, hub-лог `sessionId synced`.

**LM35. Рестарт хаба с непустой очередью топика.** `[СПЕЦ]` **P0** 🤖
- Шаги: новый топик → написать задачу (👌) → НЕ выбирая режим, рестартнуть хаб.
- Ожидаемо: сообщение доставлено после подъёма, либо честно сказано, что потеряно.
- Fail-режим: `queuedMessages`/`pendingModeChoice` — только в памяти → сообщение исчезает, 👌 остаётся
  как «принято», кнопки отвечают «Уже выбрано или устарело».
- Смотреть: реакции, чат, `hub-state.json` (там этого нет — это и есть дыра).

**LM36. Рестарт хаба посреди `/restart` или `/stop`.** `[СПЕЦ]` P1 🤖
- Ожидаемо: ложной 💀-плашки нет.
- Fail-режим: `expectedDisconnect` — в памяти → после рестарта хаба отвалившийся стаб читается как
  «оборвалась неожиданно».
- Смотреть: чат, hub-лог.

### H. Два биндинга на одну папку

**LM38. `/unbind` в A не трогает B.** `[2-ЧАТА]` P1 🤖 — ключ и tmux только A; сессия B жива, отвечает.

**LM39. Сессия заявляет чужой ключ той же папки.** `[СПЕЦ]` P2 🤖

### K. Fault-injection (кросс-группа)

**LM48. Убитый пейн при живой сессии.** `[СПЕЦ]` P1 🤖
- Шаги: `tmux kill-pane` → подождать грейс → написать сообщение.

- Fail-режим: тишина; двойное 💀; ленивый подъём не срабатывает.
- Смотреть: чат, hub-лог, `tmux ls`.

**LM49. Чужой claude ВНУТРИ нашего tmux-пейна.** `[СПЕЦ]` **P0** 🤖
- Шаги: в tmux-сессии биндинга запустить `claude` руками (без dev-канала, стаб не подключится) → `/resume`.
- Ожидаемо: хаб видит занятый пейн и отказывается печатать запуск.
- Fail-режим: `spawnSession:1926-1934` — `hasTmuxSession=true` → `typeLine` отправляет
  `cd … && TELEGRAM_BINDING_KEYS=… claude …` **как промпт в живого агента**; в чат при этом уходит
  бодрое «🪟 tmux … уже есть — набираю запуск в его активный pane» + «🚀 Возобновляю».
- Смотреть: пейн (команда как текст в инпуте/промпте), чат, `screenlog.jsonl`.

**LM50. Чужой claude в папке вне tmux.** `[СПЕЦ]` P1 🤖
- Шаги: (а) чужой claude с НАШИМ `sessionId` в argv; (б) чужой claude на другой беседе; (в) Playwright MCP
  (`@playwright/test/cli.js`) в той же папке.
- Ожидаемо: (а) блок с перечислением pid; (б) и (в) — подъём проходит (регрессии 2026-07-22).
- Смотреть: чат, hub-лог, `pgrep -af claude`.

**LM51. Битый / руками правленный `bindings.json`.** `[СПЕЦ][ЭКСКЛЮЗИВ]` **P0** 🤖
- Шаги: (а) невалидный JSON; (б) `dir` несуществующей папки; (в) лишние/пустые поля; (г) ключ неверного
  формата (`keyToTarget` кинет).
- Ожидаемо: битый файл не должен обнулять рабочие биндинги молча — нужен вопль в чат админам/лог `ERROR`.
- Fail-режим: `loadBindings` → `safeJsonParse ?? {}` → **все биндинги исчезают**, каждое сообщение
  дропается как unbound, и никто об этом не узнаёт; при (г) `keyToTarget` кидает внутри
  `reviveBoundSessions` и обрывает revive остальных.
- Смотреть: hub-лог, чат, `bindings.json`.

**LM52. Папка удалена под живым биндингом.** `[СПЕЦ]` P1 🤖
- Шаги: `rm -rf <dir>` → `/resume` (и отдельно — обычное сообщение).
- Ожидаемо: понятная ошибка «папки нет» + подсказка `/bind` заново.
- Fail-режим: «⚠️ resume не удалось: Error: tmux new-session … failed» — сырой stderr, слово `resume`
  по-английски (`hub.ts:1958`).
- Смотреть: чат, hub-лог.

**LM53. Двойная отправка команды старта.** `[СПЕЦ]` **P0** 🤖
- Шаги: `/resume` `/resume` подряд (<2с); отдельно — два быстрых тапа по одной `rs:`-кнопке; отдельно —
  `/resume` + сообщение в топик одновременно.
- Ожидаемо: ровно один claude, второй вызов видит уже идущий подъём.
- Fail-режим: `spawnSession` не имеет ни лока, ни дедупа → две команды запуска в один пейн (второй claude
  как промпт первому) или два процесса; `handleInbound` параллельно делает ленивый подъём.
- Смотреть: `tmux capture-pane`, `pgrep -af claude`, чат.

**LM54. Сессия не подключилась за 30с.** `[СПЕЦ]` P1 🤖
- Шаги: подсунуть медленный старт (`TELEGRAM_LAUNCH_CMD='sleep 60; claude …'`) → написать сообщение.
- Ожидаемо: текст говорит, где смотреть, и сообщение либо доставляется позже, либо честно помечено как
  потерянное (👌 снят).
- Fail-режим: «⚠️ Сессия не подключилась вовремя — сообщение не доставлено» (`hub.ts:2312`) и всё:
  диагностики нет, сообщение потеряно, а сессия может подняться секундой позже.
- Смотреть: чат, реакции, hub-лог, пейн.
