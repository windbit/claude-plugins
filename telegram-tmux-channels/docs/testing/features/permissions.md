# Feature: permissions

**Цель.** Когда сессия (НЕ `bypassPermissions`) вызывает тул, требующий разрешения, админ решает
из Telegram — **в топике, через picker** (Yes / Yes-and-don't-ask / No), который picker-мост
скрейпит из TUI-диалога «Do you want to proceed?».

**Код.** `detectPicker` / picker-мост. (Канальный путь `doPermissionRequest`/`pendingPermissions`
**удалён** 2026-07-18 — см. ниже.)

---

## ✅ Решено (2026-07-18): канальный 🔐-путь убран, полагаемся на picker

Решение Ромы: полагаться на picker (он в топике), слать только в прибинденный канал, никогда в ЛС.
Канальный `doPermissionRequest`/`resolvePermission`/`pendingPermissions` + их Stage 3 персист +
текстовый `y/n` + `perm:*` callback **удалены** (коммит 0a9619c). `permission_request` RPC → no-op.
Проверено e2e (non-bypass, Bash-тул): всплыл только picker в топике, без 🔐 и без висящего стейта;
тап «Yes» → команда выполнилась. Исходная находка (для истории) ниже.

### 🔴 Исходная находка: разрешение всплывало ДВУМЯ путями, канальный — невидим

При запросе разрешения в non-bypass канал-сессии срабатывают **оба** механизма на одно действие:

1. **Канальный** (`doPermissionRequest`, Stage 3): армит `pendingPermissions[request_id]` (персистится
   в `hub-state.json`) и шлёт 🔐-сообщение с «✅ Разрешить/❌ Отклонить» **в ЛС каждому админу**
   (`bot.api.sendMessage(<adminUserId>, …)`).
2. **Picker-мост** (`detectPicker`): скрейпит TUI-диалог «Do you want to proceed? 1.Yes 2.Yes-and-don't
   3.No» и рисует его кнопками **в топике**.

**Проблема:** Telegram-бот НЕ может писать в ЛС юзеру, который не нажал у бота `/start`. У тест-бота
ЛС с админом не открыт → 🔐-сообщение **молча не доставляется** (`.catch(()=>{})`), а
`pendingPermissions["waetz"]` остаётся **висеть** (персистнут, разрешается только age-out через 1ч).
Реально пользователь видит и жмёт только **picker Yes/No в топике**.

**Проверено:** non-bypass сессия (`--permission-mode default`) → «выполни ls через Bash» → агент
дошёл до `reply` тула → `hub-state.permissions["waetz"]` = запрос на `mcp__telegram__reply`,
в топике — picker (msg73 «…Do you want to proceed?»), в ЛС бота (`8907068298`) — **пусто**. Тап
«Yes» в топике разрулил через picker-путь.

**Следствия / вопрос архитектуры (нужно решение Ромы):**
- Персист `pendingPermissions` (моя Stage 3 permission-работа) на практике **вестигиальный**, если
  разрешения всегда разруливаются picker-путём в топике: канальный 🔐 не виден без открытого ЛС.
- Варианты: **(a)** убрать канальный permission-путь, полагаться на picker (проще, меньше кода);
  **(b)** слать 🔐 не в ЛС, а **в топик** (тогда виден без `/start`, и это осмысленный отдельный UX —
  явные Разрешить/Отклонить с «Подробнее» вместо сырого TUI-Yes/No); **(c)** оставить ЛС, но требовать
  от админов `/start` (хрупко). Рекомендация: (b) или (a).
- RP5 (персист пермишена через рестарт) фактически **едет на persist пикера** (RP3 ✅), пока
  канальный путь не чинится.

## Сценарии (после решения по архитектуре)
- P1: non-bypass, тул с разрешением → видимые кнопки в топике → «Разрешить» → тул выполняется.
- N1 `[СПЕЦ]`: рестарт хаба с открытым запросом → «Разрешить» после рестарта разблокирует агента.
- N2: «Отклонить» → тул не выполняется, агент получает deny.
- N3: не-админ жмёт → «Нет прав».
