# Feature: reply-fallback

**Цель.** Telegram-ход, завершившийся без единого egress-тула (`reply`/`react`/`edit`), значит агент
написал ответ в транскрипт, но забыл отправить. На turnend хаб читает финальный assistant-текст из
`.jsonl` и досылает сам («↩️ авто-досыл»). Файлы/голос/реакции — только штатным путём; фолбэк бьёт
только по реальному промаху (≈1 из 5 ходов, замерено на habebe-trader). Дедуп по `lastFallback`
(turnend может стрельнуть дважды). Персист маркеров — Stage 2.

**Код.** `forwardFallbackReply`, `armPending`/`clearPendingAnswer`, `lastFallback`/`recordFallback`,
`session-id.ts` (`lastAssistantText`/poll размера файла), `FALLBACK_MAX_CHARS`.

**Предусловие.** Контейнер, живая сессия.

---

## Позитив

**RF1. 🎯 Ход без egress → авто-досыл финального текста.** `[СПЕЦ]`
- Шаги: попросить агента ответить, НЕ вызывая `reply` (написать в терминал).
- Ожидаемо: на turnend хаб досылает «↩️ авто-досыл\n\n<финальный текст>»; текст = то, что агент
  «сказал» последним.
- Fail-режим: тишина (ответ потерян) — баг, который фича и чинит; либо преждевременный/битый текст.
- Смотреть: чат («↩️ авто-досыл»), транскрипт (финальный ассист-текст), `screenlog.jsonl`.

**RF4. Маркер `pendingAnswer` переживает рестарт.** ✅ (= RP1)
- Ссылка: `restart-persistence.md` RP1 — boot-лог `rechecking 1 pending marker`.

## Негатив / швы

**RF2. Дедуп: авто-досыл не дублируется.**
- Шаги: ход с фолбэком → turnend может стрельнуть дважды.
- Ожидаемо: один досыл (тот же текст в `lastFallback` не шлётся повторно).
- Смотреть: чат (нет дубля).

**RF3. Нет фолбэка, когда агент ОТВЕТИЛ через reply.**
- Шаги: обычный ход с `reply`.
- Ожидаемо: `clearPendingAnswer` на reply → turnend не досылает (нет двойного ответа).
- Смотреть: чат (один ответ, без «авто-досыл»).
